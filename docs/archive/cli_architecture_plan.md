# Proposal: Closing the Loop with a CLI-First Architecture

## 1. Evaluation of the Idea

The proposal to "close the loop" by moving the core analysis logic behind a CLI is a high-value strategic shift for this project.

### The Problem
Currently, the development loop relies on the browser:
1.  **Implicit Loop**: Code changes requires a Webpack HMR or full reload.
2.  **Slow Feedback**: Validating a new analysis requirement (e.g., "handle weighted medians correctly") requires clicking through the UI to load a dataset, selecting variables, and inspecting the rendered chart.
3.  **Hard to Automate**: We cannot easily write a regression test suite that verifies "weighted median of Age by Gender" matches a known truth without spinning up a full browser test (Playwright/Cypress), which is slow and brittle.

### The Solution: CLI-First
By moving the core logic to a CLI, we decouple **Correctness** from **Presentation**.
- **Correctness** (Query generation, data processing, statistical validity) is verified in milliseconds via CLI.
- **Presentation** (Charts, UI interactions) is verified in the browser.

## 2. Current Architecture & Mapping

The current codebase is actually **well-positioned** for this shift, with some refactoring needed in the Worker layer.

| Component | Current State | Readiness for CLI |
| :--- | :--- | :--- |
| **Query Generation** | `services/queryBuilder.ts` is already a collection of pure functions. | ✅ **Ready** |
| **Data Processing** | `services/analysisProcessor.ts` is pure and decoupled. | ✅ **Ready** |
| **Data Ingestion** | Logic is trapped in `services/analysisWorker.ts` (mixed with Worker boilerplate). | ⚠️ **Needs Extraction** |
| **Type Inference** | Logic is trapped in `services/analysisWorker.ts` (e.g., `detectImplicitScale`). | ⚠️ **Needs Extraction** |
| **Execution Engine** | Tied to `duckdb-wasm` in a Web Worker. | 🔄 **Replace with Native** |

### Key Insight
We do **not** need to run `duckdb-wasm` in the CLI. We can use the native `duckdb-async` or `duckdb` Node.js bindings. Since DuckDB SQL is highly portable, the SQL generated by `queryBuilder.ts` should run identically on both Native (CLI) and WASM (Browser).

## 3. Implementation Plan

The goal is to create a `scripts/analyze.ts` tool that acts as the "Headless Core".

### Phase 1: Extract Logic (Refactoring)
Moving logic out of `services/analysisWorker.ts` into pure modules makes it importable by both the CLI and the Browser Worker.

1.  **Extract Data Heuristics**:
    - Move `detectImplicitScale`, `deferVariableType`, `detectSequentialPattern` into a new `services/dataHeuristics.ts`.
    - Ensure these functions are pure (receiving data arrays, returning types).

2.  **Extract Ingestion Logic**:
    - Move the gap-filling logic and SAV metadata parsing into `services/dataIngestion.ts`.

### Phase 2: Build the CLI Harness
Create a script that simulates the Worker's job but using standard Node.js I/O.

**Prototype: `scripts/analyze.ts`**
```typescript
// Conceptual Flow
import { analyze } from './headless-core'; // The new unified entry point

const args = parseArgs();
const result = await analyze({
  dataFile: args.input, // e.g., 'test_data/dataset.sav'
  query: {
    rowVars: ['Q1', 'Q2'],
    colVar: 'Gender',
    weight: 'Weight'
  },
  format: 'json'
});

console.log(JSON.stringify(result, null, 2));
```

### Phase 3: Golden Testing
Once the CLI exists, we create a suite of "Golden Tests".
- **Input**: A known SAV file + a query definition JSON.
- **Expected**: A JSON snapshot of the results (Counts, Means, P-Values).
- **Process**: The test runner executes the CLI and diffs the output against the snapshot.

## 4. Pros and Cons

### Pros
- **Speed**: Validation drops from ~30s (manual) to ~500ms (automated).
- **Agent Autonomy**: AI agents can run `npm run analyze -- ...` to inspect data and debug logic without guessing.
- **Stability**: Refactoring `queryBuilder` becomes safe because we have regression tests covering complex SQL generation cases.

### Cons
- **Environment Parity**: Native DuckDB (CLI) might support features that WASM DuckDB (Browser) lacks, or vice versa (though rare for standard analytical SQL). We might need to pin versions carefully.
- **Refactoring Cost**: Pulling logic out of the `analysisWorker.ts` requires careful surgery to avoid breaking the existing app.
